## Detected Gaps Summary

### GAP-1: Stale Reserve State Post-Graduation
`graduate()` in FRouterV2/FRouterV3 calls `pair.transferAsset()` and `pair.transferTo()` which drain the real token balances from the pair WITHOUT updating `_pool.reserve0`, `_pool.reserve1`, or `_pool.k`. After graduation, `getReserves()` returns the last pre-graduation values while `balance()` and `assetBalance()` return 0 or near-0. Any downstream contract reading reserves post-graduation gets incorrect data. The `syncAfterDrain()` function exists and IS called in `drainPrivatePool()` (albeit via try-catch that can silently fail), but graduate() does not call it at all.

### GAP-2: syncAfterDrain Does Not Update lastUpdated
`syncAfterDrain()` mutates reserve0, reserve1, and k but leaves `_pool.lastUpdated` reflecting the timestamp of the most recent swap. After a drain, the lastUpdated field is misleading.

### GAP-3: reserve1 Is a Fake Virtual Liquidity Value
`mint()` initializes reserve1 with a fake virtual liquidity amount (e.g., 6300 VIRTUAL) that does not represent actual deposited funds. The AMM pricing formula uses this to establish a starting price. This is intentional but creates a permanent divergence between `reserve1` and `assetBalance()`. The system is documented as using "virtual" liquidity but the divergence grows as trades occur: reserve1 tracks formula-adjusted virtual amounts while assetBalance tracks real funds.

### GAP-4: k Is Frozen at mint() — Not Recomputed by swap()
`swap()` carries forward `_pool.k` unchanged — k is the initial product from `mint()` and is used as a CONSTANT by the router pricing formula. `syncAfterDrain()` DOES recompute k. After a drain (partial or full), the k stored in the pool is a new (smaller) product that DIFFERS from the constant k the router uses in its own pricing. The router reads `pair.kLast()` directly for each pricing computation, so it will use the post-drain k — this is consistent, but the k post-drain may produce very different pricing than expected if drain removes significant liquidity.

### GAP-5: taxStartTime Is Resettable by EXECUTOR_ROLE After Launch
`router.setTaxStartTime(pair, newTime)` is callable by any EXECUTOR_ROLE address at any time. After a token has already launched and its anti-sniper window is actively decaying, an EXECUTOR_ROLE operator can call this to reset taxStartTime, extending the perceived anti-sniper window OR (by setting a far-future time) blocking normal trading by making every buy appear to be a maximum-tax sniper buy. The only constraint is `_taxStartTime >= startTime`.

### GAP-6: CancelledLaunch Event in BondingV2 Emits Wrong initialPurchase
In `BondingV2.cancelLaunch()`, `_token.initialPurchase = 0` is written before the `emit CancelledLaunch(...)` call, so the event emits `0` as the initialPurchase rather than the actual amount. In BondingV5, the pattern correctly captures the value first: `uint256 initialPurchase = tokenRef.initialPurchase; tokenRef.initialPurchase = 0; emit CancelledLaunch(..., initialPurchase)`.

### GAP-7: ASYMMETRIC_BRANCH — cancelLaunch Does Not Set trading=false in BondingV2/V3/V4
BondingV5 sets `tokenRef.trading = false` in `cancelLaunch()`, providing defense-in-depth. BondingV2/V3/V4 do not. A cancelled token in V2/V3/V4 has `trading=true` and `launchExecuted=true`. The `trading` flag is checked first in `buy()` and `sell()` — if `trading=true` but `launchExecuted=true`, the second check blocks trades. However, `trading=true` post-cancel could be misleading for off-chain indexers and creates a latent risk if the guard ordering ever changes.

### GAP-8: setScheduledLaunchParams Has No Input Validation
`BondingConfig.setScheduledLaunchParams()` accepts any values with no bounds checking. Owner can set `normalLaunchFee` to MAX_UINT256, making all scheduled launches economically impossible. Or set `startTimeDelay` to 0, removing the minimum scheduling window entirely.

### GAP-9: Global gradThreshold Change Affects All In-Flight Tokens (BondingV2/V3/V4)
`setTokenParams()` in BondingV2/V3/V4 updates the global `gradThreshold` immediately. Tokens that are actively trading and approaching (but not yet hitting) the old threshold will be compared against the new threshold on the next buy. Raising the threshold causes premature graduation; lowering it delays graduation.

### GAP-10: drainPrivatePool Uses try-catch for syncAfterDrain — Silent Failure Path
If `syncAfterDrain()` reverts (e.g., on old pair contracts that don't have the function), the drain still succeeds but reserves remain stale. The router emits `PrivatePoolDrained` regardless. The comment acknowledges this is intentional for backward compatibility, but it means the reserve0/reserve1 values for old-contract pairs permanently reflect pre-drain quantities.

### GAP-11: volume24H Is Never Written During Trades
`data.volume24H` is initialized to 0 at preLaunch and never updated in `buy()`, `sell()`, or any other function. Only `data.lastUpdated` is conditionally updated. The 24h volume tracking infrastructure is incomplete — the field exists in the struct but is dead code from the perspective of on-chain writes.
